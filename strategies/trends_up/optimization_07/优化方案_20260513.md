## 一、为什么需要 `optimization_07`

### 1.1 optimization_06 的结论

`optimization_06` 在 2026-05-12 完成了完整的实验验证并已封档。核心发现：

1. **opt04 纯基线（git HEAD `19d0d20`）仍然可复现**：2025 年 +62.24%，与 opt_02 历史文档一致。
2. **Phase 1B（working tree）2025 年仅 +32.0%**，比 opt04 低约 30 个百分点。
3. **opt_06 设计的"A1 暴露增强"和"A2 候选弹性恢复"全部无效**——两者均无法缩小 2025 的 30% 差距。
4. **根本原因**：working tree 的 33 个 hunks 中包含大量未 gated 的代码改动，即使在 `ENTRY_SCORE_V2=False` 时仍显著改变策略行为（2025 从 +62% 降至 +14%）。

### 1.2 PTrade 真实环境的额外问题

`optimization_04` 目录下保存了在 PTrade 真实环境中运行的日志和交易记录：

| 指标 | 2024 | 2025 |
|------|------|------|
| 总交易次数 | 85 | 39 |
| 买入次数 | 44 | 22 |
| 卖出次数 | 41 | 17 |
| **资金不足放弃买入** | **333** | **265** |
| 硬止损触发 | 24 | 4 |
| 时间止损触发 | 8 | — |

**核心问题**：策略在首日建仓后迅速满仓（5/5），后续因卖出释放资金过少，"可用资金不足 10000 元"触发极其频繁（每年 265-333 次），导致策略长期处于"无法买入"的瘫痪状态。PTrade 真实环境的交易频率远低于 SimTradeLab 本地回测。

### 1.3 本轮要解决的两个问题

1. **结构性退化**：working tree 中哪些未 gated 改动导致了 2025 从 +62% 降至 +32%？
2. **资金效率**：10 万资本 + 5 只满仓 = 资金锁死。在 PTrade 真实环境中，这个问题比本地回测更严重。

## 二、代码状态盘点

### 2.1 两个关键版本

| 版本 | 位置 | 2025 | 说明 |

### 2.2 33 hunks 变更分类

working tree 相对 git HEAD 的改动分为以下类别：

| 类别 | 文件区域 | 是否 gated | 预估影响 |
|------|---------|:----------:|---------|
| A. ENTRY_SCORE_V2 选股评分 | SelectionAgent, ConfigManager | ✅ 是 | 排序变化 |
| B. Phase 1B 准入分层 | SelectionAgent._get_b_min_score_for_mode 等 | ✅ 是 | B 级收缩 |
| C. 候选质量差距约束 | SelectionAgent.select() | ✅ 是 | 弱机会少买 |
| D. ROTATION_V2 | ExecutionAgent, ConfigManager | ✅ 是（已关闭） | 无 |
| E. EXIT_STAGE_V1 | PositionAgent, ConfigManager | ✅ 是（已关闭） | 无 |
| F. **find_weakest_position 改用 holding_score** | ExecutionAgent | ❌ **否** | **高** |
| G. reentry_cooldown 机制 | ExecutionAgent._is_reentry_cooldown_active | ❌ **否** | 中 |
| H. 增强报表 + run_id 隔离 | ReportAgent, initialize | ❌ 否 | 无（不影响交易） |
| I. A1/A2 实验代码 | ConfigManager, ExecutionAgent, SelectionAgent | ✅ 是（已关闭） | 无 |

**F 类是最可疑的未 gated 改动**：`find_weakest_position` 从"找亏损最多的持仓"改为"找 holding_score 最低的持仓"，直接影响轮换卖出对象的选择。

## 三、核心假设

1. **2025 的 30% 差距主要由 F 类（未 gated 改动）导致**，而非 ENTRY_SCORE_V2 本身（因为 A2 激进版禁用 ENTRY_SCORE_V2 后 2025 仍为 +32%，未恢复到 +62%）。

2. **如果从 opt04 出发，仅添加经过验证的 gated 防御模块**，可以在保留 opt04 进攻弹性的同时，获得 Phase 1B 的熊市防守能力。

3. **PTrade 真实环境的"资金不足"问题**源于：首日建仓 3 只 → 满仓 5 只 → 卖出释放资金不足 1 万 → 无法买入。需要调整初始建仓节奏和最小买入门槛的匹配关系。

## 四、总体方案：从 opt04 出发，逐层叠加

### 4.1 方案原则

1. **基准是 opt04 纯基线（git HEAD）**，不是 Phase 1B
2. **每次只叠加一类改动**，立即跑双年回测验证
3. **只保留对 2024 有正面/中性效果且不明显伤害 2025 的改动**
4. **对导致退化的改动，设计替代方案而非直接丢弃**
5. **同时解决 PTrade 资金锁死问题**

### 4.2 实施阶段

```
Phase 1：资金效率修复（PTrade 兼容）
  → 降低首日建仓激进程度 + 调整最小买入门槛
  → 基线对照：opt04 + 资金修复 vs opt04 纯基线

Phase 2：逐层叠加 gated 防御模块
  → 2A: ENTRY_SCORE_V2 选股评分（仅排序，无准入限制）
  → 2B: Phase 1B 准入分层（B 级上限 + 分模式 MIN_SCORE_B）
  → 2C: 候选质量差距约束
  → 每层独立验证双年回测

Phase 3：修复未 gated 退化改动
  → 3A: find_weakest_position 对比（holding_score vs profit_only）
  → 3B: reentry_cooldown 必要性评估
  → 对确认为退化的改动，设计替代方案

Phase 4：组合验证
  → 将所有通过验证的模块组合
  → 最终双年验证
```

## 五、Phase 1：资金效率修复

### 5.1 问题诊断

PTrade 2025 年日志显示：首日（1月2日）买入 3 只，次日（1月3日）再买 3 只 → 5 只满仓。之后每次卖出释放资金仅 ~8000-16000 元，但 `actual_cash_to_use <= 10000` 门槛导致大量"资金不足"。

根因链：
```
10万资本 → 首日建仓3只(每只~1.6-2.4万) → 持仓5/5满仓
→ 卖出释放资金 < 1万门槛 → 无法买入新标的
→ 旧持仓持续亏损 → 更多硬止损 → 进一步消耗资金
→ 策略陷入"只能卖不能买"的瘫痪状态
```

### 5.2 修复方案

| 参数 | opt04 当前值 | 修复值 | 说明 |
|------|------------|--------|------|
| 首日建仓上限 | 3 只 | **2 只** | 留更多现金缓冲 |
| 最小买入金额门槛 | 10000 元 | **5000 元** | 卖出释放的小额资金也能复用 |
| 现金保留比例 | 10% | **15%** | 始终保留更多现金 |

### 5.3 验收标准

1. PTrade 2025 年"资金不足"从 265 次降至 < 100 次
2. 全年交易次数从 39 次提升至 > 60 次
3. 2025 收益率不低于 opt04 纯基线的 90%（即 ≥ +56%）
4. 2024 收益率不低于 0%（即不亏损）

### 5.4 实施步骤

1. 从 git HEAD 创建 `optimization_07` 分支
2. 修改 `buy_new()` 中的首日建仓逻辑
3. 修改 `actual_cash_to_use` 的最小门槛
4. 修改 `cash_reserve_ratio`
5. 2025Q1 冒烟 → 2025 全年 → 2024 全年

## 六、Phase 2：逐层叠加 gated 防御模块

### 6.1 Phase 2A：ENTRY_SCORE_V2 选股评分（仅排序增强）

**目标**：仅添加 RS60 + 负向惩罚的排序增强，不添加任何准入限制。

**从 opt04 的改动**：
- 新增 `FEATURE_FLAGS["ENTRY_SCORE_V2"] = True`
- 新增 `ENTRY_SCORE_V2` 配置（但关闭准入分层：`MAX_NEW_B_POSITIONS = None`，`MIN_SCORE_B_*` 使用默认值 0.45）
- 新增 `compute_entry_features()` 和 `score_entry_v2()`
- 在 `select()` 中接入排序增强

**禁止改动**：
- 不改 `MAX_NEW_B_POSITIONS`（保持 None，无 B 级上限）
- 不改 `MIN_SCORE_B_SIDEWAYS/DOWNTREND`（保持默认 0.45）
- 不改 `ALLOW_2OF4_B_*`（保持全部 True）
- 不改 `MIN_CANDIDATE_SCORE_GAP`（不启用质量差距）
- 不改 `find_weakest_position`

**验收**：
- 2025 不低于 opt04 纯基线的 90%（≥ +56%）
- 2024 不显著低于 opt04 纯基线

### 6.2 Phase 2B：准入分层（B 级上限 + 分模式 MIN_SCORE_B）

**前置条件**：Phase 2A 通过。

**目标**：在排序增强基础上，添加 Phase 1B 的准入分层。

**改动**：
- `MAX_NEW_B_POSITIONS = 2`
- `MIN_SCORE_B_SIDEWAYS = 0.52`，`MIN_SCORE_B_DOWNTREND = 0.60`
- `ALLOW_2OF4_B_IN_SIDEWAYS = False`，`ALLOW_2OF4_B_IN_DOWNTREND = False`

**验收**：
- 2024 不低于 +10%（准入分层应改善熊市表现）
- 2025 不低于 opt04 纯基线的 80%（≥ +50%）

### 6.3 Phase 2C：候选质量差距约束

**前置条件**：Phase 2B 通过。

**目标**：在弱机会日主动减少买入。

**改动**：
- `MIN_CANDIDATE_SCORE_GAP = 0.08`
- `CANDIDATE_GAP_SKIP_REDUCE = 1`

**验收**：
- 2024 不低于 +12%
- 2025 不低于 Phase 2B 水平

## 七、Phase 3：修复未 gated 退化改动

### 7.1 Phase 3A：find_weakest_position 对比实验

**背景**：working tree 中 `find_weakest_position` 从"找亏损最多"改为"找 holding_score 最低"。这是未 gated 的改动，可能是 2025 退化的主要原因。

**实验设计**：
1. 在 Phase 2 通过后的代码上，临时将 `find_weakest_position` 回退到 opt04 版本（profit_only）
2. 对比 holding_score 版本 vs profit_only 版本的双年回测
3. 如果 profit_only 显著优于 holding_score → 保留 profit_only
4. 如果 holding_score 更优 → 保留 holding_score
5. 如果各有优劣 → 考虑分市场模式使用不同策略

### 7.2 Phase 3B：reentry_cooldown 必要性评估

**背景**：working tree 新增了 `_is_reentry_cooldown_active`，卖出后 10 天内禁止回补同一标的。

**实验设计**：
1. 对比启用/禁用 cooldown 的双年回测
2. 评估 cooldown 是否减少了无效轮换还是阻碍了正常回补

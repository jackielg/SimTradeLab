## 一、问题诊断：为什么资金利用率极低

### 1.1 数据事实

从 opt04 交易明细可直接量化：

| 指标 | 2024 | 2025 |
|------|:----:|:----:|
| 总交易笔数 | 86 | 40 |
| 买入笔数 | ~43 | ~20 |
| 卖出笔数 | ~43 | ~20 |
| 年均持仓数 | 4.8只 | 5.1只 |
| 首日建仓后满仓天数占比 | ~90% | ~85% |

**全年平均换手率极低**：2025年仅约20次买入（含首日建仓），意味着平均每月不到2次新买入机会。对于一个10万元账户，这意味着约85%的时间资金完全锁死在现有持仓中。

### 1.2 根因链

```
MA20/MA120 止损过于迟钝
    → 盈利股不卖（"让利润奔跑"变成"让利润睡觉"）
    → 亏损股等到 -12% 硬止损才卖（资金锁死数月）
    → 卖出释放资金远小于建仓需求
    → 每日选股选出候选但无法买入（"资金不足"）
    → 策略实际上变成 Buy & Hold
```

核心矛盾：**MA20 在趋势市中有效（2025 +62%），但它不产生 turnover。** 策略捕获了趋势，但没有在趋势减弱时及时退出并重新部署资本。

### 1.3 opt07 吊灯止损为什么"零触发"

opt07 中测试的吊灯止损参数：
- ATR(10) multiplier = 3.0（推测，未明确记录）
- 峰值回撤无 profit 限制
- 结论："零触发，MA20 已覆盖"

根因：multiplier 3.0 太宽松。在 A 股 1 分钟级别回测中，ATR(10)×3.0 的通道宽度远大于 MA20 的偏移距离，永远不会在 MA20 之前触发。

**opt08 的关键区别**：使用更短的 ATR 周期（22日）+ 更紧的乘数（1.5~2.5），让吊灯止损在 MA20 之前触发，主动释放资本。

## 二、核心假设

1. **MA20 止损过于迟钝是资金锁死的根本原因**。当前策略唯一切实的退出是硬止损 -12%（亏损股）和极少数的止盈（downtrend L1/L2）。uptrend/sideways 中止盈设为 9.99（禁用），MA20/MA120 在趋势市中几乎从不触发。

2. **引入更敏感的 ATR 吊灯止损可以在不明显伤害收益率的前提下大幅提升资金周转**。ATR 止损在"趋势减弱但尚未反转"时退出，释放的资本可立即部署到新的强势候选。

3. **去除 MAX_POSITIONS 限制并配合单笔最低交易额**，允许将释放的小额资本汇聚后再投资，避免"有候选但资金不足"。

4. **基于波动率（ATR）的动态止损比固定均线止损更适合 A 股的高波动特征**。

## 四、模块详细设计

### 模块 A：ATR 吊灯止损（主力出口）⭐⭐⭐⭐⭐

#### A.1 吊灯止损公式

```
trailing_stop = highest_close_since_entry - ATR(period) × multiplier
exit_signal: close < trailing_stop
```

#### A.2 分市场模式参数

| 参数 | uptrend | sideways | downtrend |
|------|:-------:|:--------:|:---------:|
| ATR period | 22 | 22 | 14 |
| multiplier | 2.5 | 2.0 | 1.5 |
| 最低盈利门槛（启用条件） | 盈利 > 1×ATR | 盈利 > 0.5×ATR | 盈利 > 0 |
| 确认 bar 数 | 1 | 1 | 1 |

**设计理由**：
- uptrend multiplier=2.5：够宽以让大趋势充分发展（约为 2.5 倍日均波动），但比 MA20 更敏感
- sideways multiplier=2.0：在震荡市中更快锁定利润，产生周转
- downtrend multiplier=1.5：极紧，亏损头寸迅速退出；盈利头寸稍有回撤即锁定

#### A.3 与 MA20 的关系

吊灯止损是**主出口**，MA20/MA120 降级为**兜底安全网**：

| 出口 | 优先级 | 用途 |
|------|:------:|------|
| 硬止损 -12% | 1（最高） | 防止灾难性亏损 |
| 时间止损 30d/-5% | 2 | 防止无限期锁死 |
| ATR 吊灯止损 | 3（主出口） | 主动管理，释放资本 |
| MA20/MA120 | 4（兜底） | 深跌保护 |

#### A.4 盈利门槛保护

吊灯止损仅在持仓有一定盈利后才启用（防止亏损期过早退出）：

```
启用条件：highest_close_since_entry > cost_price × (1 + entry_threshold)
其中 entry_threshold = ATR / cost_price（约 2%~5%）
```

**关键**：这对于 downtrend 模式尤其重要——在 downtrend 中 multiplier=1.5，但只有盈利 > 0 才启用吊灯。亏损头寸走硬止损或时间止损。这避免了 opt04 中"硬止损是唯一出口"的问题，但不创造假信号。

#### A.5 盈利锁定增强

当持仓盈利超过一定水平后，在吊灯止损基础上叠加保本线：

```
if profit_ratio > 0.15:  # 盈利超过 15%
    trailing_stop = max(trailing_stop, cost_price × 1.03)  # 至少保本+3%
if profit_ratio > 0.30:  # 盈利超过 30%
    trailing_stop = max(trailing_stop, cost_price × 1.10)  # 锁定至少 10%
```

### 模块 B：去除持仓上限，单笔最低交易额 ⭐⭐⭐⭐

#### B.1 新规则

```
移除：MAX_POSITIONS = 5
新增：MIN_TRADE_AMOUNT = 10000  # 每笔最低 10,000 元
新增：TARGET_POSITION_SIZE = 20000  # 每只目标仓位 20,000 元
```

#### B.2 仓位分配逻辑

```
available_cash = context.portfolio.cash

# 计算可开新仓数量
max_new_positions = max(0, available_cash // MIN_TRADE_AMOUNT)

# 每只分配
position_size = min(TARGET_POSITION_SIZE, available_cash // max(1, len(candidates)))

# 实际买入量（向下取整到 100 股）
actual_amount = min(position_size, available_cash)
shares = (actual_amount / price) // 100 * 100
```

#### B.3 与候选排名的协调

```python
# 按 entry_score 降序，依次买入
for stock, entry_score in target_stocks:
    if available_cash < MIN_TRADE_AMOUNT:
        break
    # 分配资金（S级可多分配，A/B级按基准）
    alloc_mult = 1.5 if grade == "S" else 1.0
    buy_amount = min(TARGET_POSITION_SIZE * alloc_mult, available_cash)
    buy_amount = max(MIN_TRADE_AMOUNT, buy_amount)
    # 执行买入
    ...
    available_cash -= buy_amount
```

**关键优势**：不再有"5只满仓，资金锁死"的问题。当 ATR 止损释放资本后，同日（或次日）即可部署到新的强势候选。

### 模块 C：盘中 ATR 监测（15分钟级别）⭐⭐⭐

#### C.1 设计动机

日线级别的 ATR 吊灯止损在趋势反转时可能反应滞后 1~2 天。引入 15 分钟级别的辅助监测，在日内捕捉趋势崩溃。

#### C.2 实现

```python
# 15分钟K线 ATR(14) 快速监测
df_15m = DataCache.get_15m_data(stock, 480)  # 约20天
atr_15m = compute_atr(df_15m, 14)

# 日内吊灯止损（更紧）
intraday_mult = 1.0  # 仅为日线的 50%
intraday_stop = highest_close_15m - atr_15m * intraday_mult

if close_15m < intraday_stop and profit_ratio > 0.05:
    # 盈利>5%但在15分钟级别破位 → 卖出
    order_target(stock, 0)
```

#### C.3 约束条件

- 仅在 14:00 后触发（避免盘中假信号过早卖出）
- 仅在盈利 > 5% 时启用（避免亏损头寸日内恐慌卖出）
- 每只股票每天最多触发一次（避免反复交易）

### 模块 D：止盈参数微调 ⭐⭐

#### D.1 当前问题

opt04 在 uptrend/sideways 中将 L1/L2 止盈设为 9.99（完全禁用），这导致 "让利润奔跑" 变成了 "让利润睡觉"。

#### D.2 调整方案

| 参数 | opt04 | opt08 | 说明 |
|------|:-----:|:-----:|------|
| uptrend L1 | 9.99（禁用）| **0.60 / 20%** | 盈利60%时卖出20% |
| uptrend L2 | 9.99（禁用）| **1.00 / 20%** | 盈利100%时再卖20% |
| sideways L1 | 9.99（禁用）| **0.40 / 25%** | 盈利40%时卖出25% |
| sideways L2 | 9.99（禁用）| **0.70 / 25%** | 盈利70%时再卖25% |
| downtrend L1 | 0.10 / 30% | **不变** | |
| downtrend L2 | 0.20 / 30% | **不变** | |

**设计理由**：在 uptrend/sideways 中引入高位止盈，但不卖全部（保留大部分仓位继续奔跑）。L1/L2 共释放约40%仓位，剩余60%由 ATR 吊灯止损管理。这样既有部分利润落袋，释放资本，又不完全踏空超级牛股。

### 7.3 向量化 + 批量优化

#### 7.3.1 `select()` 选股向量化

**当前**：per-stock 循环，逐个调 `get_daily_data()` → 对 400 只候选循环 400 次：

```python
# 当前 (高开销)
for stock in stock_list:  # 400只
    df = DataCache.get_daily_data(stock, 65)  # 每次 cache lookup
    # ... 逐个计算均线/MACD/条件判定
```

**优化**：在 `select()` 之前批量预加载 + 向量化条件判定：

```python
# 优化后
# 1. 批量预加载（已有 preload_daily_data，但需确保覆盖 select 所需）
DataCache.preload_daily_data(stock_list, 70)

# 2. 预计算全量 close 序列到 numpy array（一次完成）
close_matrix = {stock: df["close"].values for stock, df in preloaded.items()}

# 3. 部分条件用 numpy 向量化判定
for stock in stock_list:
    # 从预计算矩阵直接取值，零开销
    ...
```

#### 7.3.2 盘后报表批量查询

**当前**：`generate_daily_reports()` 和 `generate_profit_loss_report()` 各自独立调用 `get_daily_data()`，对同一只股票可能重复查询 2~3 次。

**优化**：盘后单次批量拉取：

```python
# after_trading_end 中一次性查询
active_stocks = [s for s, p in positions.items() if p.amount > 0]
DataCache.preload_daily_data(active_stocks, 65)  # 批量预加载
# 后续 generate_daily_reports / generate_profit_loss_report 从缓存读取
```

#### 7.3.3 `_filter_by_price_and_trend()` 向量化

**当前**：per-stock 循环 + per-stock `get_daily_data()`：

```python

    df = DataCache.get_daily_data(stock, 65)  # 每只查一次
    # close, ma20, ma60, cond1/2/3 判定
```

**优化**：利用 `build_watchlist` 已预加载的 65 日缓存（`preload_daily_data` 已调用过），内联改为从缓存直接读取：

```python
today_str = g.current_date_str
for stock in candidates:
    cache_key = f"{stock}_65_False_{today_str}"
    df = DataCache._cache.get(cache_key)  # 从缓存直接读，零API调用
    if df is None:
        df = DataCache.get_daily_data(stock, 65)  # fallback
    # 条件判定逻辑不变
```

### 7.4 缓存增强

#### 7.4.1 ATR 日级缓存

```python
# 新增：ATR 值按 (stock, date, period) 缓存
g._atr_cache = {}  # key: f"{stock}_{date}_{period}", value: float

@staticmethod
def compute_atr_cached(df, period):
    cache_key = f"{stock}_{today_str}_{period}"
    if cache_key in g._atr_cache:
        return g._atr_cache[cache_key]
    atr = PositionAgent.compute_atr(df, period)
    g._atr_cache[cache_key] = atr
    return atr
```

#### 7.4.2 `compute_holding_score` 分钟级去重

**当前**：`check_intraday_exit` 每 15 分钟执行一次 → 每只持仓股各计算一次 `compute_holding_score` → 相同日期下重复计算。

**优化**：已有日级缓存 `_holding_score_daily_cache`，确认生效即可。在 `check_intraday_exit` 的 MA 止损路径中跳过 holding_score 计算（MA 止损不需要它）。

#### 7.4.3 选股 day-only 标记

**当前**：ATS 在 09:35 选股 + 14:30 再次选股 = 每天 2 次完整选股漏斗。

**优化**：`build_watchlist` 添加日级缓存标记，一天只执行一次：

```python
if getattr(g, "_watchlist_built_date", "") == g.current_date_str:
    return  # 跳过，今天已执行
g._watchlist_built_date = g.current_date_str
```

### 7.5 `check_intraday_exit` 频率优化

**当前**：每 15 分钟触发一次（09:35 ~ 14:30，约 20 次/天）。每次遍历所有持仓股，对每只股票执行完整的 MA 止损、保本线、止盈检查。

**优化**：
1. 止损检查从 15 分钟截流改为 30 分钟截流（减少 50% 执行次数）
2. 每只股票每 30 分钟只做一次快速 MA 比较（取最新 bar 的 close 和 MA 值）
3. ATR 吊灯止损仅在 14:00 和 14:30 两次检查（日终更可靠）

```python
# 优化后频率
if m % 30 == 0:  # 30分钟截流 (vs 15分钟)
    # 快速 MA 比较（无 holding_score 计算）
    ...
if h == 14 and m in (0, 30):  # 14:00 + 14:30 ATR检查
    PositionAgent.check_atr_chandelier_exit(context)
```

**优化**：

## 八、风险分析

### 8.1 ATR 吊灯止损过紧的风险

**风险**：multiplier 太小 → 频繁退出盈利股 → 踏空大趋势 → 2025 收益率暴跌。

**缓解**：
- uptrend 使用 multiplier=2.5（相对宽松），让超级牛股充分奔跑
- 设置最低盈利门槛（盈利 > 1×ATR 才启用），避免"刚盈利就被震出"
- Phase 1 首先测试 multiplier 参数（2.0 / 2.5 / 3.0），找到不伤害 2025 的平衡点

### 8.2 无限持仓导致过度分散的风险

**风险**：去除了 MAX_POSITIONS=5 → 资金分配到太多低质量候选 → 单只仓位过小 → 手续费侵蚀收益。

**缓解**：
- MIN_TRADE_AMOUNT=10000 天然限制了极端分散
- TARGET_POSITION_SIZE=20000 保持每只有意义的仓位
- 现有选股评分系统确保只有高分候选被买入
- 候选本身就稀缺（日均仅 3~8 只入选），不会出现"买 20 只"的情况

### 8.3 2024 年进一步恶化的风险

**风险**：2024 是熊市/震荡市，ATR 止损在下跌趋势中可能频繁触发 → 反复"买入→止损→买入"的 whip-saw → 2024 收益率比 opt04 的 -0.97% 更差。

**缓解**：
- downtrend multiplier=1.5 非常紧，快速退出
- downtrend 中早盘买入关闭（MORNING_ENTRY_ENABLED=False），减少暴露
- 但承认 2024 年趋势跟踪策略的固有限制（opt07 已验证这是策略类型的上限）
- 如果 Phase 1 测试显示 2024 恶化严重，可为 downtrend 单独调宽 multiplier 或关闭 ATR 止损

## 九、预期效果推演

### 9.1 基于 opt04 交易记录的模拟推演

以 opt04 2025 年最成功的交易为例（603200.SS +172%）：

| 阶段 | opt04 行为 | opt08 行为 |
|------|-----------|-----------|
| 买入 1/21 @26.19 | 买入 800 股 | 买入 800 股 |
| 到 3 月中旬涨至 ~45（+72%） | 持有不动（uptrend 无止盈）| 吊灯止损跟踪最高价 |
| 3 月下旬回调至 ~40（从高点 -11%）| 持有不动 | ATR(22)×2.5 吊灯止损触发，卖出 |
| 释放资本 ~32,000 | — | 部署到新候选 |
| 后继续涨至 ~70 | 继续持有（完整捕获）| 已退出，错过后续 |

**结论**：ATR 吊灯止损会在超级牛股的中期回调中退出，可能错过部分后续涨幅。但释放的资本可部署到其他牛股。**净效果取决于替代投资的质量。**

### 9.2 核心赌注

**这个优化的核心赌注是**：在当前选股系统能稳定选出强势股的前提下，更高的资金周转率 × 略低的单笔收益率 > 低周转率 × 高单笔收益率。

opt04 的 603200.SS +172% 是杰出的单笔交易，但它只占资金的约 20%。如果同期把资本周转了 3 次，每次赚 15~20%，总收益可能更高。
